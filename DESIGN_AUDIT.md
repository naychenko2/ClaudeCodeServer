# Дизайн-аудит ClaudeHomeServer — соответствие макетам Claude Design

Сквозной аудит: насколько **реализация** соответствует **задумке дизайнера** в проекте Claude Design `52adb1f7-312b-4f25-8c47-2bccfca9df94`. Сверялись не с собственными токенами кода, а с тем, что спроектировано в макетах — где задумка реализована иначе, упрощена или отсутствует.

## Метод

- **Макеты прочитаны напрямую** из Claude Design (DesignSync): 3 интерактивных прототипа (мобильный 392×844, планшет 1180×820, десктоп 1440×884), каталог состояний чата (stream-json, разделы A–F), доска пограничных состояний (18 фреймов), исследование композера (6 состояний × 2 раскладки), дизайн-система v1.0.
- **Реализация:** ~30 живых Playwright-скриншотов (`audit/screenshots/`) во всех трёх режимах + код.
- **5 параллельных сверок** — по одной на прототип/доску. Все подтвердили доступ к макетам и работали с исходниками, не по памяти.

## Главный вывод

**Статика соответствует, динамика — нет.** Базовые экраны (логин, проекты, сессии, файлы, просмотр кода/markdown/изображений) в целом следуют макетам, а местами реализация даже **богаче задумки** (см. раздел «Сверх макета»). Но **агентная динамика — лента чата (поток stream-json) — реализована фрагментарно**: целые задуманные классы состояний отсутствуют как визуальные блоки. Плюс несколько готовых, но не подключённых или обеднённых элементов, где данные уже есть, а UI их не показывает.

## P1 — критичные расхождения с задумкой

| # | Зона | Задумано в макете | Реализовано | Нюанс |
|---|---|---|---|---|
| **MD4 / MA2** | Чат | Закреплённый чек-лист **«ПЛАН N/M»** (TodoWrite): выполнено ✓ / текущее (пульс) / ожидает | TodoWrite падает в общий `tool_use` → **пустая техническая строка** | Самый узнаваемый элемент UX Claude Code отсутствует |
| **MD2** | Чат | Итог хода — «квитанция»: **стоимость, токены (in/out), накопление за сессию**, длительность | «✓ N шагов · Xс» | `usage` **уже приходит** в `ResultMessage`, но теряется при маппинге в `ChatItem`; `total_cost_usd` бэкенд не извлекает — дешёвый выигрыш |
| **MD3** | Чат | 4 разных **ошибочных футера** хода (`max_turns`/`during_execution`/`max_budget`/`structured_retries`) с действиями | `subtype` доходит, но в UI лишь ✓/✗ | При упоре в лимит — серое «✗ N шагов» без причины и кнопки |
| **MD1** | Чат | **Ошибки генерации** с разбивкой: `rate_limit` (авто-ретрай), `auth` («переавторизоваться»), `billing`, `max_tokens` («продолжить»), `server` | Одна плашка «⚠ текст» + «Повторить» | Для банковского контура auth/billing — ключевые состояния |
| **MD5** | Чат | Карточка **AskUserQuestion** с выбором вариантов A/B и обратным каналом | Прилетает безликим `tool_use`, выбрать нельзя | Модель не может задать уточняющий вопрос — ход зависнет/пойдёт по догадке |
| **MA1** | Сессии | Карточка сессии: превью + **число сообщений** + **относительное время** | Только имя, превью и бейдж модели | `messageCount`, `updatedAt` **есть в модели** — чистое упущение UI |
| **MC1** | Планшет | Центр отдаёт **либо чат, либо файл во всю ширину** (взаимоисключающе) | Файл fullscreen **+ постоянный док чата снизу** | Добавленная конструкция, которой в планшетном макете нет — согласовать |

## Сквозные зоны расхождений

### ① Состояния ленты чата недореализованы — главная зона
Сверка каталога stream-json (`Chat States Gallery`, разделы A–F) показала, что обработан лишь костяк. Что задумано, но **отсутствует или частично**:

- **Нет:** TodoWrite-план (MD4), ошибки генерации с разбивкой (MD1), AskUserQuestion (MD5), различение ошибок итога хода (MD3), стриминг аргументов инструмента (C2), типизация инструментов — Edit-diff / Bash-терминал / MCP-значок (MD8), компакция контекста (MD11), вложенность Task-субагентов (MD9), отказы разрешений в футере (D2), «Копировать/Повторить» под ответом (MD10).
- **Частично:** карточка init сессии — только модель+режим, без cwd/MCP/числа инструментов (A1); итог хода без стоимости/токенов (MD2); тело результата инструмента не показывается, только «готово/ошибка» (MD6); запрос разрешения без `tool_input` и без «всегда разрешать» (MD7); размышление без `redacted`-варианта; replay/synthetic не различаются, нет «Дослать» после прерывания (MD12).
- **Совпадает:** стриминг+markdown ответа, сворачиваемое размышление, `file_changed`-карточка (Открыть/Откатить), базовый запрос разрешения, empty-state с подсказками, офлайн-композер, привязка tool_result→tool_use.

> Многое дёшево: `usage` и `subtype` уже на проводе, `NewSessionDialog` уже написан — нужно дотянуть UI/маппинг, а не строить с нуля.

### ② Готово, но не подключено / обеднено
- **`NewSessionDialog`** (имя + первый промпт + режим + модель) полностью соответствует макету, но **нигде не вызывается** — «+ Новый чат» создаёт пустую сессию сразу (MA6).
- **Карточка проекта:** в макете — **число сессий**, в реализации — дата обновления (MA13).
- **Авто-создание сессии** при входе в проект скрывает задуманное пустое состояние «Ещё нет сессий» (MA14).

### ③ Просмотр файла упрощён
- Нет **нумерации строк** в коде и diff (макет рисует gutter) — MA3 / MB2.
- Diff — крашеный `<pre>` без gutter и без структурированных заголовков ханков `@@` (MB3).

### ④ Композер: подмена паттернов
- Режимы Авто/План/Спросить задуманы **выпадающим меню с описаниями и галочкой**, реализованы скрытым **циклическим кликом** по пилюле без подсказок (ME2).
- Кнопка режима — системные **цветные эмодзи** `⚡📋❓` вместо терракотовой **штриховой иконки** из эталона, ломает монохромную иконографику (ME1).

### ⑤ Планшетная раскладка
Макет: рельс **288px (фиксированный)** + центр под чат **или** файл целиком. Реализация: рельс перетаскиваемый 220–520 + файл fullscreen со свёрнутым доком чата снизу (MC1–MC3). Это осознанная адаптивная конструкция — нужно либо принять как продуктовое решение, либо вернуть к макету.

### ⑥ Мелкие тексты и знаки
Плейсхолдер сервера `https://example.com` вместо макетного `https://code.acme.dev` (MA10); вкладки рельса «Чаты/Файлы» вместо макетных «Сессии/Файлы» (MB4, расходится и с протоколом); «1 шагов» без склонения; «симв.» вместо «ток.» в размышлении (MA12); иконка логина-терминал вместо макетной «искры» (MA11).

## Реализовано СВЕРХ макета (расширения, не деградация)

Сверка зафиксировала и обратное — где реализация **богаче** задумки: полноценный **markdown-рендер** ответов (таблицы, gfm), **офлайн-синхронизация** файлов («облако»), **скачивание** файлов, **поиск** по проектам и по файлам, empty-state чата «Чем помочь?» с подсказками, **редактирование проекта**, перетаскиваемые сплиттеры и ширина рельса, **фокус-режим** с доком чата. Это плюсы — при сведении к макету их стоит сохранить.

## Рекомендованный порядок работ

1. **Состояния чата** — самый ценный и недостающий пласт. Начать с дешёвых, где данные уже на проводе: итог хода со стоимостью/токенами (MD2, пробросить `usage`/`total_cost_usd`), TodoWrite-план (MD4), `tool_input` + «всегда разрешать» в разрешении (MD7), тело результата инструмента (MD6). Затем — ошибки генерации (MD1), AskUserQuestion (MD5), типизация инструментов (MD8).
2. **Подключить `NewSessionDialog`** и обогатить карточки сессии (число сообщений/время) и проекта (число сессий) — данные уже есть (MA1, MA6, MA13).
3. **Нумерация строк + структурированный diff** в просмотре файла (MA3, MB2, MB3).
4. **Композер-режимы** — меню с описаниями вместо цикл-клика; штриховые иконки вместо эмодзи (ME1, ME2).
5. **Планшетную раскладку** согласовать с дизайнером (принять адаптив или вернуть к макету) (MC1).
6. Мелкие тексты/иконки + (вторично) дисциплина токенов `design.ts` (`audit/01–04`).
