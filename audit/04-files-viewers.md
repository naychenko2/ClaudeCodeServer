# Дизайн-аудит 04 — Файловый менеджер и просмотрщики

Область: дерево файлов, поиск, FileViewer (шапка/код/изображения/diff/бинарные),
просмотрщики Markdown / PDF / Docx / Xlsx, ProjectSyncToggle.
Метод: Read кода + анализ скриншотов (`da-ws-*.png`). Код приложения не менялся.

## Эталон (дизайн-система)

Источник истины — `frontend/src/lib/design.ts` (объекты `C`, `BTN`).

Фоны: `bgMain #F4F0E8`, `bgPanel #EDE7DC`, `bgCard #FBF8F2`, `bgWhite #FFFFFF`, `bgSelected #E8E1D4`.
Текст: `textHeading #2A251F`, `textPrimary #39332B`, `textSecondary #756B5E`, `textMuted #9A8F7E`.
Акцент (ОСНОВНОЙ, оранжевый): `accent #D97757`, `accentLight #F4ECE1`, `accentMuted #EAD3C5`.
Границы: `border #E0D7C8`, `borderLight #E8E1D4`.
Статусы: `success #5E8B4E` (bg `#E9F1E8`, text `#3F7A4F`), `warning #C9923E`, `danger #B4452F` (bg `#FBF1EC`), `info #3E7CA6` (bg `#E7EFF5`).
Diff: `diffAddBg #E8F5E9` / `diffAddText #1B5E20`, `diffRemBg #FFEBEE` / `diffRemText #B71C1C`.
Шрифты: PT Serif (заголовки), Hanken Grotesk (UI), JetBrains Mono (код, пути файлов, имена файлов).
Кнопки `BTN`: primary — фон `accent`, текст `#FBF8F2`, radius 11; secondary — фон `bgPanel`; ghost — рамка `border`, radius 9.
Стили — только inline-объекты. Цвета — только из палитры (новые не вводить).
Брейкпоинты: mobile < 768, tablet 768–1100, desktop ≥ 1100.

Общий вывод: визуально компоненты близки к эталону и аккуратны. Главные проблемы —
функциональные/информационные (поиск без путей; сырой JSON с `\uXXXX`), плюс
системная россыпь хардкод-цветов мимо токенов и расхождение «зелёный/красный»
diff-статистики с палитрой статусов. Ниже — по находкам.

### [F1] Поиск возвращает неразличимые дубли — результаты без пути/папки
- **Severity:** P1
- **Режим:** все
- **Где:** `FileExplorer.tsx:365-366` (рендер `searchResults.map(... renderFileRow(entry, 0))`), `renderFileRow` 281-287; скрин `da-ws-search-results-desktop.png`, `da-ws-files-tablet.png`, `da-ws-files-mobile.png`
- **Проблема:** запрос «design.ts» вернул 4 строки `design.ts`, абсолютно идентичных — показывается только `entry.name`, без относительного пути/родительской папки. Пользователь не может отличить `frontend/src/lib/design.ts` от копий в `publish/…`, `da-…` и т.п. Клик — лотерея: непонятно, какой именно файл откроется. Для проекта, где имена файлов часто повторяются (`index.ts`, `page.tsx`, `README.md`), результаты поиска практически бесполезны.
- **Предложение:** в режиме поиска под именем файла выводить относительный путь до папки вторичным цветом (`textMuted #9A8F7E`, шрифт JetBrains Mono, fontSize 11). Бэкенд в `entry.path` путь уже отдаёт — нужно лишь показать `dirname(entry.path)`. Опционально — подсветить совпадение запроса в имени (bold/`accent`). Это снимает неоднозначность и заодно объясняет «дубли».

### [F2] JSON-файлы показываются сырым текстом с `\uXXXX` — кириллица нечитаема
- **Severity:** P1
- **Режим:** все
- **Где:** `FileViewer.tsx:464-472` (ветка `SyntaxHighlighter` для не-markdown); скрин `da-ws-file-tablet.png`, `da-ws-file-mobile.png` (`history.json`)
- **Проблема:** `.json` с экранированными юникод-последовательностями (`"Код…"`) отображается как есть. Весь русский текст превращается в стену `\u04xx`, файл нечитаем. На мобильном это занимает целый экран бессмыслицы. Подсветка JSON есть, но содержимое всё равно непонятно человеку.
- **Предложение:** для `.json` (и `.json5`) перед показом распарсить и переформатировать с `JSON.stringify(JSON.parse(content), null, 2)` — `JSON.parse` сам декодирует `\uXXXX` в нормальные символы, отступы выровняются. При ошибке парса — fallback на сырой текст (как сейчас). Опционально — тумблер «сырой / форматированный». Это превращает нечитаемый дамп в нормально читаемый документ. Замечание: бэкенд лучше отдавал бы файл уже как UTF-8 строку, но даже клиентский re-parse решает проблему отображения.

### [F3] Diff-статистика `+N/-N` в шапке — цвета мимо палитры статусов
- **Severity:** P2
- **Режим:** все
- **Где:** `FileViewer.tsx:280-281`
- **Проблема:** счётчики используют `#27AE60` (зелёный) и `#C0392B` (красный) — это значения из «общей» палитры ТЗ, но НЕ из активных токенов проекта. В `design.ts` статусы другие: `success #5E8B4E`, `danger #B4452F`. То есть «зелёный плюс» в шапке ярче и насыщеннее, чем success-цвет в остальном приложении (бейджи статусов сессий и т.п.) — визуальный разнобой оттенков «успеха/опасности».
- **Предложение:** заменить на токены: `+` → `C.successText #3F7A4F` (или `C.success`), `-` → `C.dangerText #B4452F`. Привести к единому семейству зелёного/красного.

### [F4] Diff-вьюер: цвета захардкожены, не сверены с diff-токенами
- **Severity:** P3
- **Режим:** все
- **Где:** `FileViewer.tsx:482-483` (фон/текст строк diff)
- **Проблема:** строки красятся литералами `#E8F5E9`/`#1B5E20`/`#FFEBEE`/`#B71C1C`. Значения СОВПАДАЮТ с `C.diffAddBg/diffAddText/diffRemBg/diffRemText` — то есть визуально верно, но дублируется магия. Контекстные строки красятся в `#555` — это не токен (ближайший `textSecondary #756B5E`).

Мелкий нюанс: классификация строки по `startsWith('+')`/`'-'` затрагивает и заголовки ханков `@@`/`+++`/`---` — стоит исключить `+++`/`---` (как уже сделано в подсчёте `diffStats` строк 241-242).

### [F5] Шапка FileViewer на мобильном перегружена — до 7 контролов в ряд, тач-таргеты < 44px
- **Severity:** P2
- **Режим:** mobile
- **Где:** `FileViewer.tsx:258-398`; скрин `da-ws-file-mobile.png`
- **Проблема:** в одну строку умещаются «‹ Файлы», имя файла (ellipsis), pill «Файл/Diff», «Править», облако, корзина, «✕». На 390px при открытом редактировании и наличии diff («Откатить») будет ещё теснее. Иконочные кнопки имеют `padding: 4` при иконке 14-15px → реальный тач-таргет ~22-23px, вдвое меньше рекомендованных 44px. Корзина и «закрыть» стоят вплотную — высок риск промаха (случайное удаление вместо закрытия особенно неприятно). На скрине «Править» — крупная primary-кнопка, а рядом мелкие иконки без подписей — иерархия читается, но плотность на грани.
- **Предложение:** на мобильном (1) увеличить тач-зону иконок до min 40-44px (`padding: 10-12`, либо фиксированный `minWidth/minHeight: 40`); (2) развести «удалить» и «закрыть» (разнести, либо delete убрать в overflow-меню «⋯»); (3) при нехватке места — вторичные действия (облако/синхронизация, корзина, expand) сворачивать в overflow-меню, оставив в строке только назад · имя · Файл/Diff · Править. Это снизит риск ошибочных тапов и разгрузит шапку.

### [F7] Россыпь хардкод-цветов мимо токенов `C` во всех компонентах

Не вводить новые произвольные значения.

- **Severity:** P3

### [F9] Подсветка кода: getLanguage маппит больше языков, чем зарегистрировано в FileViewer
- **Severity:** P2
- **Режим:** все
- **Где:** `FileViewer.tsx:32-48` (registerLanguage) vs `getLanguage.ts:1-51` (EXT_MAP)
- **Проблема:** `getLanguage` возвращает языки `powershell` (`.ps1`/`.psm1`), `ruby` (`.rb`), `kotlin` (`.kt`), `cpp`/`c` (`.cpp/.c/.h/.hpp`), `toml`, `less`, `xml` (`.xml/.svg`), `json5`, `swift`, `graphql`, `dockerfile`, `protobuf`, `hcl`, `lua`, `r`, `php`, `makefile`. Но в FileViewer зарегистрированы только: tsx, typescript, javascript, jsx, csharp, json, markdown, css, scss, python, go, rust, java, bash, yaml, sql, markup. Для незарегистрированных языков (например `.ps1` — а это PowerShell-проект на Windows, `.xml`, `.toml`, `.cpp`) `react-syntax-highlighter/prism-light` не найдёт грамматику → подсветки не будет (текст монохромный) или возможна ошибка/варнинг в консоли. На скрине `da-ws-file-code-desktop.png` TS подсвечивается корректно — проблема только для непокрытых расширений.
- **Предложение:** либо зарегистрировать недостающие prism-грамматики (как минимум `powershell`, `xml`/`markup` уже есть как `markup` — но getLanguage возвращает строку `'xml'`, а зарегистрирован `'markup'` → не совпадает!), либо в `getLanguage` маппить на уже зарегистрированные имена (`xml`/`svg` → `'markup'`). Сейчас `.svg`/`.xml`/`.html` → `'xml'`/`'html'`, а зарегистрирован только `'markup'` и `'css'` — несостыковка имён. Свести EXT_MAP и набор registerLanguage к одному списку, чтобы не было «тихих» провалов подсветки.

### [F11] Дерево: иконки типов различимы, но папки и `.dotfiles` без иконки типа
- **Severity:** P3
- **Режим:** все
- **Где:** `FileExplorer.tsx:29-49` (EXT_META), `266-280`; скрин `da-ws-files-desktop.png`
- **Проблема:** (1) цветные бейджи расширений (`ts`, `img`, `git`, `md`…) — хорошее, различимое решение, на скриншоте читаются. Но fallback для неизвестного расширения берёт первые 3 буквы ext или `•` (строка 48) — для файлов без расширения (`LICENSE`, `Makefile`) бейдж будет `•`, а `.gitignore` показан как `git` (по правилу `split('.').pop()` → ext=`gitignore` → нет в карте → первые 3 буквы `git`) — случайно сработало, но `.gitattributes` дал бы `git` тоже, а `.npmrc` → `npm`. Логика «расширение = после последней точки» для dotfiles даёт неожиданные бейджи. (2) Папки все с одинаковой оранжевой иконкой — нормально, но спец-папки (`.git`, `.claude`) ничем не выделены.
- **Предложение:** (1) для имён, начинающихся с точки и без второй точки (`.gitignore`, `.npmrc`), трактовать имя целиком как «расширение» осознанно или показывать нейтральный значок; добавить в EXT_META частые dotfiles. Мелочь, но влияет на консистентность. (2) Папки — оставить как есть (приемлемо).

### [F12] Дерево: длинный список (десятки PNG) — нет виртуализации, горизонтальный скролл вместо ellipsis
- **Severity:** P3
- **Режим:** все
- **Где:** `FileExplorer.tsx:254-264` (`width: 'max-content', minWidth: '100%'`), `355` (контейнер `overflow: auto`); скрин `da-ws-files-desktop.png`
- **Проблема:** (1) строки растут по содержимому (`width: max-content`) → длинные имена дают горизонтальный скролл всей панели (видно по нижнему скроллбару на десктоп-скрине). Это намеренное решение (комментарий в коде), но горизонтальная прокрутка дерева — спорный UX: имя файла обычно лучше обрезать с ellipsis, а полное показывать в title/tooltip. (2) Десятки строк рендерятся без виртуализации — для текущего демо ок, но на больших репозиториях (тысячи файлов в раскрытой папке) будет лагать. Контекст задачи отмечает, что «забитость PNG — артефакт», но архитектурно длинный список не оптимизирован.
- **Предложение:** (1) рассмотреть `textOverflow: ellipsis` + `title={name}` вместо горизонтального скролла (консистентнее с FileViewer, где имя файла уже обрезается ellipsis — строка 273). (2) Виртуализацию — отметить как технический долг, не блокер для демо.

### [F13] Просмотрщики PDF/Docx/Xlsx: состояния error/loading консистентны, но текст ошибки не предлагает действий
- **Severity:** P3

## Резюме

- **P1: 2** — F1 (поиск без путей, неотличимые дубли), F2 (сырой JSON с `\uXXXX`, кириллица нечитаема).
- **P2: 4** — F3 (diff-статистика мимо палитры), F5 (мобильная шапка перегружена, тач-таргеты < 44px), F7 (россыпь хардкод-цветов), F9 (подсветка кода: рассинхрон getLanguage и registerLanguage).
- **P3: 15** — F4, F6, F8, F10, F11, F12, F13, F14, F15, F16, F17, F18, F19, F20, F21.

**Топ-3:**
1. **F1** — поиск возвращает идентичные строки без путей; для проекта с повторяющимися именами файлов поиск фактически неюзабелен (на трёх скриншотах четыре одинаковых `design.ts`).
2. **F2** — JSON показывается как стена `\uXXXX`, кириллица нечитаема; лечится клиентским `JSON.parse`/`stringify`.
3. **F9** — `getLanguage` обещает подсветку языков (PowerShell, XML, C++, TOML…), для которых грамматики не зарегистрированы и/или имена не совпадают (`xml` vs `markup`) → тихие провалы подсветки на не-TS файлах.

Общая оценка: визуально раздел аккуратен и близок к макетам (дерево, шапка viewer, markdown, изображение читаются хорошо). Системные слабости — информативность поиска, обработка JSON, дисциплина токенов (десятки литералов мимо `C`/`BTN`, расхождение diff-статистики и модалок с системными значениями) и покрытие подсветки кода.
