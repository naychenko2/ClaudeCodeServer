# Дизайн-аудит: Аутентификация и Проекты

Область аудита: экраны входа (`LoginPage.tsx`) и списка проектов (`ProjectListPage.tsx`), включая диалоги «Новый проект», «Редактировать проект», «Удалить проект».

## Эталон (дизайн-система)

Беру за эталон `frontend/src/lib/design.ts` (объекты `C` и `BTN`) и секцию «Дизайн-система» в `CLAUDE.md`.

**Цвета (`C`):**
- Фоны: `bgMain #F4F0E8`, `bgPanel #EDE7DC`, `bgCard #FBF8F2`, `bgWhite #FFFFFF`, `bgSelected #E8E1D4`
- Текст: `textHeading #2A251F`, `textPrimary #39332B`, `textSecondary #756B5E`, `textMuted #9A8F7E`
- Акцент: `accent #D97757`, `accentLight #F4ECE1`, `accentMuted #EAD3C5`
- Границы: `border #E0D7C8`, `borderLight #E8E1D4`
- Статусы: `success #5E8B4E`, `danger #B4452F`, `info #3E7CA6`, `warning #C9923E` (+ их Bg/Text варианты)

**Кнопки (`BTN`):** primary `accent` фон / radius 11 / padding 11×20 / fontSize 14 / weight 600; secondary `bgPanel` фон; ghost border `border` / radius 9; danger `danger` фон / radius 9.

**Радиусы:** кнопки/поля 9–11px (эталон BTN). **Шрифты:** PT Serif — заголовки, Hanken Grotesk — UI, JetBrains Mono — код/пути. **Стили:** только inline-объекты.

**Сквозное правило:** цвета и параметры должны браться из `C`/`BTN`, а не хардкодиться литералами. Любой литерал вне палитры — находка.

## Сквозные находки (обе страницы)

### [A1] Токены design.ts не используются — всё захардкожено литералами
- **Severity:** P2
- **Режим:** все
- **Где:** `LoginPage.tsx` (весь файл), `ProjectListPage.tsx` (весь файл) — ни один из них не импортирует `C`/`BTN` из `lib/design.ts`
- **Проблема:** оба экрана полностью на строковых литералах цветов и параметров кнопок. Даже там, где значение совпадает с токеном (`#D97757`, `#F4F0E8`, `#E0D7C8`, `#756B5E`, `#9A8F7E`, `#2A251F`, `#B4452F`), оно продублировано вручную. Это нарушает правило «цвета из дизайн-системы», создаёт дрейф значений (см. A2) и делает невозможным централизованное изменение темы. `design.ts` существует именно как single source of truth, но фактически мёртв на этих экранах.
- **Предложение:** импортировать `import { C, BTN } from '../lib/design'` и заменить литералы на токены: `#F4F0E8 → C.bgMain`, `#EDE7DC → C.bgPanel`, `#E0D7C8 → C.border`, `#FFFFFF → C.bgWhite`, `#2A251F → C.textHeading`, `#756B5E → C.textSecondary`, `#9A8F7E → C.textMuted`, `#D97757 → C.accent`, `#B4452F → C.danger`, `#5E8B4E → C.success`. Кнопки диалогов свести к `BTN.primary`/`BTN.secondary`/`BTN.danger` (с поправкой радиусов, см. A2).

### [A2] Цвета вне палитры дизайн-системы
- **Severity:** P2
- **Режим:** все
- **Где:** `LoginPage.tsx:101` (`#1A1612`), `:217` (`#5C5246`), `:197` (`#D8CFBE`), `:273-284` (`#FFF0EE`, `#F5C6BF`, `#B03010`); `ProjectListPage.tsx:14-19` плитки (`#E7F0E8/#3F7A4F`, `#E6EEF5`, `#FBEBE0/#C2693B`, `#F2E6F0/#8E4A82`), `:323/:346` (`#B0A697`), `:372/:411` (`#D0C6B4`), `:379/:389` (`#BE5536`)
- **Проблема:** в палитре `C` нет ни одного из перечисленных значений. Примеры: заголовок логина `#1A1612` темнее токена `textHeading #2A251F`; подпись тоггла `#5C5246` — нет в палитре (рядом `textPrimary #39332B`/`textSecondary #756B5E`); фон выключенного тоггла `#D8CFBE` не из палитры; блок ошибки логина (`#FFF0EE`/`#F5C6BF`/`#B03010`) полностью самописный, хотя в `C` есть `dangerBg #FBF1EC`/`danger #B4452F`/`dangerText #B4452F`; акцент «Добавить проект» `#BE5536` и `#C2693B` в плитке — отдельные оттенки оранжевого мимо `accent #D97757`. Это размывает дизайн-систему и даёт визуально несогласованные оттенки.
- **Предложение:** заголовок → `C.textHeading`; подпись тоггла → `C.textPrimary` или `C.textSecondary`; блок ошибки → `background: C.dangerBg`, `border: 1px solid C.accentMuted` (или производный от danger), `color: C.dangerText`; «Добавить проект» и иконку-плюс → `C.accent`; плитки проектов — свести к парам на базе токенов статусов (`successBg/successText`, `infoBg/info`, `accentLight/accent`) либо завести явные токены палитры плиток в `design.ts`. Серый иконок действий `#B0A697` → `C.textMuted`.

## Экран входа (LoginPage)

Скрины: `da-login-desktop.png`, `da-login-tablet.png`, `da-login-mobile.png`, `da-login-mobile-filled.png`.

### [A3] Радиусы полей и кнопки превышают эталон (9–11px)
- **Severity:** P3
- **Режим:** все
- **Где:** `LoginPage.tsx:129` и `:157` (поля `borderRadius: 13`), `:234` (главная кнопка `borderRadius: 14`), `:80` (логотип `borderRadius: 15`)
- **Проблема:** правило дизайн-системы — радиусы кнопок/полей 9–11px (`BTN.primary` = 11). Здесь поля 13px, главная кнопка 14px. Само по себе мягко и аккуратно выглядит на скринах, но расходится с остальным UI (поиск проектов `borderRadius: 12`, карточки 16 — тоже разнобой) и с эталоном.
- **Предложение:** привести к шкале дизайн-системы: поля и кнопка `borderRadius: 11` (как `BTN.primary`); если хочется чуть мягче для крупной формы — единое значение 12 на всех полях/кнопках этого экрана, но согласованно, не 13/14/15 вразнобой.

### [A4] Кнопка «Подключиться» disabled слабо отличима от активной
- **Severity:** P2
- **Режим:** все
- **Где:** `LoginPage.tsx:236-237` (`opacity: isDisabled ? 0.7 : 1`), скрины `da-login-desktop.png`/`da-login-mobile.png` (disabled, поля пустые) vs `da-login-mobile-filled.png` (активная)
- **Проблема:** заблокированное состояние = просто `opacity 0.7` при сохранённой акцентной заливке и тени `0 4px 14px rgba(217,119,87,0.3)`. На desktop/tablet/mobile-пустых скринах кнопка выглядит как полноценно кликабельная (насыщенный оранжевый, объёмная тень). Пользователь будет жать неактивную кнопку и не понимать, почему ничего не происходит. Разница с активной (`filled`) — минимальная.
- **Предложение:** для disabled убрать тень (`boxShadow: 'none'`), снизить заливку до `C.accentMuted`/`C.bgSelected` либо `opacity 0.45`, добавить `cursor: not-allowed` (он уже есть). Цель — однозначно «выключенный» вид.

### [A5] Поля логина без видимого label — только плейсхолдер
- **Severity:** P2
- **Режим:** все
- **Где:** `LoginPage.tsx:140-149` (поле сервера), `:168-177` (поле ключа)
- **Проблема:** назначение полей передаётся только плейсхолдером (`https://example.com`, `sk-ant-••••`). При заполнении плейсхолдер исчезает, и подпись «что это за поле» теряется (классическая проблема placeholder-as-label, бьёт по доступности и по возвращению к форме). Для поля API-ключа это особенно критично: ввод маскируется, контекст теряется полностью. Иконки (глобус/замок) — слабая замена текстовой метке для скринридера.
- **Предложение:** добавить видимые метки над полями («Адрес сервера», «API-ключ») размером 12–13px `C.textSecondary`, либо как минимум `aria-label` на оба `input`. Метка-сверху согласуется с диалогами проектов, где у полей тоже только плейсхолдеры (см. A12).

### [A6] Контраст плейсхолдеров и подсказок ниже комфортного
- **Severity:** P3
- **Режим:** все
- **Где:** `LoginPage.tsx:329` (подпись «Соединение шифруется» `#9A8F7E` 12.5px), плейсхолдеры полей (цвет браузерный поверх белого), подзаголовок `:115` (`#756B5E`)
- **Проблема:** `textMuted #9A8F7E` на белом/`bgMain` для текста 12–12.5px даёт контраст около 2.7:1 — ниже WCAG AA (4.5:1) для мелкого текста. На скринах подпись под кнопкой и плейсхолдеры читаются с напряжением. То же на странице проектов (даты, путь).
- **Предложение:** для информативного текста ≤13px использовать `C.textSecondary #756B5E` (контраст ~4.3:1, почти AA) вместо `textMuted`. `textMuted` оставить только для совсем второстепенных подписей крупнее или для иконок.

### [A7] Состояние ошибки скрывает кнопку «Подключиться» и сбрасывает подпись
- **Severity:** P3
- **Режим:** все
- **Где:** `LoginPage.tsx:223` (`{!isError && (<button .../>)}`), `:269-325` (блок ошибки с двумя кнопками вместо неё), `:328` (подпись тоже под `!isError`)
- **Проблема:** при ошибке основная кнопка «Подключиться» полностью исчезает, заменяясь парой «Изменить ключ / Повторить». Макет ошибки не предоставлен в скринах, но логика подменяет первичное действие, и подпись «Соединение шифруется» пропадает — высота формы скачет. Это работоспособно, но переключение «было одно действие — стало два других» дезориентирует; «Повторить» функционально дублирует «Подключиться».
- **Предложение:** показывать блок ошибки *над* неизменной кнопкой «Подключиться» (которая и есть «повторить»), оставив только вторичную «Изменить ключ» как ghost-кнопку. Подпись про шифрование не убирать. Так макет стабильнее и понятнее.

### [A8] Кнопки в состоянии ошибки — обводка 1.5px вне токенов
- **Severity:** P3
- **Режим:** все
- **Где:** `LoginPage.tsx:295` (`border: '1.5px solid #D97757'`, radius 11), `:288-323`
- **Проблема:** «Изменить ключ» сделана как outline-кнопка с обводкой 1.5px цвета accent и accent-текстом — этого варианта нет в `BTN` (есть `ghost` с `1px solid border`). Дробная толщина 1.5px и accent-обводка вводят новый тип кнопки вне дизайн-системы.
- **Предложение:** свести к `BTN.ghost` (1px `C.border`, текст `C.textSecondary`) либо к `BTN.secondary`. Парную «Повторить» — к `BTN.primary`. Толщину границы держать 1px.

### [A9] Тоггл «Сохранить ключ» — нестандартный неинтерактивный для скринридера
- **Severity:** P3
- **Режим:** все
- **Где:** `LoginPage.tsx:181-220` (`div onClick`, без `role`/`aria-checked`, без реального `input[type=checkbox]`)
- **Проблема:** переключатель собран из `div` с `onClick`. Нет семантики чекбокса/свитча, не фокусируется с клавиатуры, недоступен для скринридера. Тач-таргет самого тоггла 42×25px — по высоте < 44px (хотя кликом ловится вся строка с подписью, что смягчает).
- **Предложение:** добавить `role="switch"`, `aria-checked={saveKey}`, `tabIndex={0}` и обработчик клавиатуры (Space/Enter), либо скрытый нативный `input`. Это снимает и доступность, и тач-таргет (вся строка кликабельна).

### [A10] Перенос заголовка на mobile разрывает «Claude Code» / «Server»
- **Severity:** P3
- **Режим:** mobile
- **Где:** `LoginPage.tsx:96-109`, скрин `da-login-mobile.png` (заголовок «Подключение к / Claude Home Server»)
- **Проблема:** на desktop/tablet перенос идёт после «Claude Code» (строка 2 = «Server»), на mobile — после «к» (строка 2 = «Claude Home Server»). Точка переноса пляшет между режимами; на mobile «Подключение к» висит коротким огрызком. Не критично, но снижает аккуратность ключевого заголовка.
- **Предложение:** либо сократить заголовок на mobile до «Claude Home Server» с подзаголовком «Подключение к серверу», либо зафиксировать перенос через явный `<br/>`/неразрывные пробелы, чтобы «Claude Home Server» держалось целиком.

## Список проектов (ProjectListPage)

Скрины: `da-projects-desktop.png`, `da-projects-tablet.png`, `da-projects-mobile.png`.

### [A11] Иконки «Редактировать»/«Удалить» — тач-таргет 26×26px (< 44px) и почти невидимы
- **Severity:** P1
- **Режим:** mobile (и все)
- **Где:** `ProjectListPage.tsx:316-318` и `:341-343` (`width: 26, height: 26`), цвет `#B0A697`, `gap: 4`, скрин `da-projects-mobile.png`
- **Проблема:** кнопки действий 26×26px — почти вдвое меньше минимума 44px для тача. На mobile они стоят вплотную (`gap: 4`) у правого края узкой карточки, серым `#B0A697` на белом (контраст ~2.3:1) — еле различимы и в них трудно попасть пальцем, легко промахнуться между «редактировать» и «удалить» (а удаление — деструктивно). На скрине `da-projects-mobile.png` иконки сливаются с обрезанным путём проекта рядом. Это реальная проблема юзабилити и доступности именно на мобильном.
- **Предложение:** увеличить тач-зону до 40–44px (`width/height: 40`, иконка внутри 18px, центрирование сохранить), увеличить `gap` до 6–8px, цвет иконок поднять до `C.textSecondary` с hover-фоном `C.bgHover`/`C.bgSelected`. На очень узком экране рассмотреть скрытие действий в меню «···» или свайп, чтобы не теснить контент.

### [A12] Имя проекта обрезается раньше времени из-за иконок-действий
- **Severity:** P2
- **Режим:** mobile
- **Где:** `ProjectListPage.tsx:282-303` (текстовый блок `flex:1, minWidth:0`) + `:310-362` (блок действий справа), скрин `da-projects-mobile.png` («Демо · ClaudeCodeSer…», путь «c:\Sources\ClaudeCodeServ…»)
- **Проблема:** на mobile и имя, и путь режутся многоточием, при этом справа постоянно занимают место две иконки действий + плитка 50px слева. Полезная ширина под текст мала, поэтому даже короткое «Демо · ClaudeHomeServer» не помещается. Путь обрезается с *конца* (`c:\Sources\ClaudeCodeServ…`) — теряется самая значимая часть (имя конечной папки), которая как раз отличает проекты `\backend` от `\frontend`.
- **Предложение:** на mobile (1) убрать иконки действий из строки в свайп/меню (см. A11), освободив ширину; (2) для пути использовать обрезку с середины или с начала (`direction: rtl` + `text-align: left`, либо middle-ellipsis), чтобы был виден хвост `…\backend`. Имя можно разрешить в 2 строки на mobile.

### [A13] Карточки проектов: радиусы и тени вне дизайн-системы
- **Severity:** P3
- **Режим:** все
- **Где:** `ProjectListPage.tsx:257` (`borderRadius: 16`), `:260` (`boxShadow: '0 2px 8px rgba(60,50,35,0.04)'`), `:267` (плитка `borderRadius: 14`), `:215` (поиск `borderRadius: 12`)
- **Проблема:** карточки 16px, плитка 14px, поиск 12px, кнопки диалогов 13px (см. A14), модалки 20px — пять разных радиусов на одном экране, ни один не совпадает с эталонной кнопочной шкалой 9–11px. Тень `rgba(60,50,35,0.04)` — почти невидима (на скринах карточки держатся только границей `#E8E1D4`), не соответствует и описанной в CLAUDE.md тени карточек `0 1px 4px rgba(60,50,35,0.08)`.
- **Предложение:** ввести согласованную шкалу: поля/кнопки 11, карточки 12–14 (одно значение), модалки 16. Тень карточек привести к `0 1px 4px rgba(60,50,35,0.08)` либо убрать вовсе, оставив только границу `C.border`.

### [A14] Кнопки и поля в диалогах — радиусы/цвета мимо BTN
- **Severity:** P2
- **Режим:** все
- **Где:** `ProjectListPage.tsx:99-136` (`btnCancel`/`btnAccent`/`btnDanger`: `borderRadius: 13`, `padding: 14`), `:84-97` (`dialogInput borderRadius: 12`), `:101-102` (`btnCancel` фон `#EDE7DC`, цвет `#756B5E`), скрины `da-addproject-desktop.png`/`da-addproject-mobile.png`
- **Проблема:** кнопки диалогов почти повторяют `BTN.primary/secondary/danger`, но с радиусом 13 (эталон 11/9) и квадратным `padding: 14` вместо `11px 20px`. То есть это «почти токены», переписанные вручную с отклонениями — классический дрейф. На скрине `da-addproject-desktop.png` кнопки заметно круглее, чем primary-кнопки в остальном приложении.
- **Предложение:** заменить `btnAccent → BTN.primary`, `btnCancel → BTN.secondary`, `btnDanger → BTN.danger` (с добавлением `flex: 1`). Поля диалогов — `borderRadius: 11`, `border: 1px solid C.border`.

### [A15] Поиск проектов: введённый текст будет серым (цвет плейсхолдера применён к value)
- **Severity:** P2
- **Режим:** все
- **Где:** `ProjectListPage.tsx:235` (`color: '#9A8F7E'` на самом `input`)
- **Проблема:** цвет `textMuted` навешен на сам `<input>`, а не только на плейсхолдер. Значит реально введённый пользователем запрос будет отображаться приглушённым `#9A8F7E` (контраст ~2.7:1 на белом) — как будто это всё ещё плейсхолдер. Пользователь не получает чёткой обратной связи о вводе.
- **Предложение:** цвет вводимого текста — `C.textHeading`/`C.textPrimary`; для плейсхолдера задать `::placeholder { color: C.textMuted }` (через `<style>`-тег или CSS-переменную, раз inline `::placeholder` недоступен). Поле поиска на скринах пустое, поэтому баг не виден, но проявится при вводе.

### [A16] Пустое состояние — слабое, без призыва и иллюстрации
- **Severity:** P2
- **Режим:** все
- **Где:** `ProjectListPage.tsx:395-399` (текст «Нет проектов. Создайте первый выше.»), а также `:368-391` (dashed-кнопка «Добавить проект» всё равно рендерится)
- **Проблема:** при полном отсутствии проектов экран показывает: поиск, dashed-кнопку «Добавить проект», и под ней мелкую серую строку `#9A8F7E` 14px «Нет проектов. Создайте первый выше.». Это бедное empty-state: нет заголовка, нет иконки/иллюстрации, нет внятного объяснения ценности, призыв («выше») косвенный и завязан на позицию dashed-кнопки. Указанные «пустые» скрины (`da-projects-empty-desktop.png`, `da-projects-empty-mobile.png`) фактически показывают экран логина (см. A19), так что эталон empty-state не задан — тем важнее сделать его по-человечески.
- **Предложение:** при пустом списке показать центрированный блок: иконка/иллюстрация (например тот же терминальный знак, что в логотипе), заголовок PT Serif «Пока нет проектов», подпись `C.textSecondary` «Добавьте папку с кодом, чтобы начать», и одну заметную primary-кнопку «Добавить проект» (`BTN.primary`). Dashed-кнопку в этом состоянии лучше заменить полноценной CTA, а не дублировать.

### [A17] Состояние «ничего не найдено» оставляет dashed-кнопку «Добавить проект»
- **Severity:** P3
- **Режим:** все
- **Где:** `ProjectListPage.tsx:368-391` (кнопка всегда в конце списка) + `:400-404` (сообщение «Ничего не найдено по запросу…»)
- **Проблема:** при поисковом запросе без результатов список пуст, но dashed-кнопка «Добавить проект» всё равно показывается (она вне условия `filtered.length`), а под ней — сообщение «Ничего не найдено». Получается «Добавить проект» + «Ничего не найдено» рядом — логически странно: добавить *что*, текущий запрос как имя? Призыв сбивает с толку при фильтрации.
- **Предложение:** скрывать dashed-кнопку, когда `search !== '' && filtered.length === 0`; в сообщении предложить действие «Очистить поиск» (сброс `search`) и/или «Создать проект “{search}”», передав запрос как имя в диалог.

### [A18] Аватар «AD» и серверный URL — жёстко зашитые/неинформативные
- **Severity:** P3
- **Режим:** все
- **Где:** `ProjectListPage.tsx:203` (`AD` хардкод), `:205` (`label={serverUrl || 'localhost'}`), бейдж `:174-206`, скрины `da-projects-*.png` («AD · localhost»)
- **Проблема:** инициалы аватара «AD» захардкожены строкой, не выводятся из реального пользователя/ключа — на скринах у всех всегда «AD». Весь бейдж — это кнопка выхода (`onClick={onLogout}`), но визуально это выглядит как индикатор статуса, а не «выйти»: нет иконки выхода, нет подсказки. Риск случайного логаута при тапе по тому, что кажется статус-чипом.
- **Предложение:** либо явно обозначить действие (иконка «выход» / тултип «Выйти» / `aria-label`), либо вынести логаут в отдельный пункт меню, а бейдж оставить чисто статусным. Инициалы — генерировать или убрать, не зашивать «AD».

### [A19] «Пустые» скрины проектов фактически показывают экран логина
- **Severity:** P2
- **Режим:** desktop, mobile
- **Где:** артефакты `da-projects-empty-desktop.png`, `da-projects-empty-mobile.png`
- **Проблема:** оба файла, заявленные как «список проектов (пусто)», содержат не пустой список проектов, а форму входа (идентичны `da-login-desktop.png`/`da-login-mobile.png`). Это не баг приложения, а ошибка набора скриншотов: реальное empty-state приложения (A16) осталось неотснятым, его невозможно сверить с макетом. Фиксирую как находку по артефактам, чтобы переснять.
- **Предложение:** переснять пустое состояние списка проектов (после логина, с пустым `projects.json`) и заменить эти два файла. До тех пор empty-state оценивать только по коду (см. A16).

### [A20] Мобильный диалог «Новый проект» — центрированный overlay, а не bottom-sheet
- **Severity:** P3
- **Режим:** mobile
- **Где:** `ProjectListPage.tsx:138-153` (`overlayStyle` center, `modalStyle width: 400`), `:411` («ручка» drag-handle сверху), скрин `da-addproject-mobile.png`
- **Проблема:** на скрине `da-addproject-mobile.png` диалог визуально прижат к низу и имеет drag-handle (полоску `40×4`), как у bottom-sheet, но код центрирует overlay по вертикали и фиксирует `width: 400` — на узком экране (390px) это даёт почти-полноэкранную карточку по центру с «ручкой», которая ничего не «тянет» (нет swipe-to-dismiss). Семантика handle вводит в заблуждение: жест перетаскивания не работает. Несогласованность паттерна между «выглядит как шит» и «ведёт себя как модалка».
- **Предложение:** на mobile (< 768) реализовать настоящий bottom-sheet: `alignItems: 'flex-end'`, `width: 100%`, `borderRadius: 16px 16px 0 0`, и либо включить swipe-to-dismiss для handle, либо убрать handle на desktop/tablet (где он сейчас тоже есть и не нужен — см. `da-addproject-desktop.png`, полоска висит над центрированной модалкой без смысла).

### [A21] Поля диалогов без меток, плейсхолдер «Путь к папке» с трекингом
- **Severity:** P3
- **Режим:** все
- **Где:** `ProjectListPage.tsx:416-427` (диалог создания), `:445-456` (диалог редактирования), `dialogInput` `:84-97`, скрины `da-addproject-desktop.png`/`da-addproject-mobile.png`
- **Проблема:** у полей диалога только плейсхолдеры «Название» / «Путь к папке», без видимых меток — при заполнении контекст исчезает (та же проблема, что A5). Плейсхолдер «Путь к папке» наследует `fontFamily: JetBrains Mono` и на скринах выглядит разреженно («П у т ь  к  п а п к е»), что читается хуже обычного UI-плейсхолдера. Поле «Название» — обычный текст, но тоже моноширинное, хотя название — не код.
- **Предложение:** добавить метки над полями; поле «Название» сделать шрифтом `Hanken Grotesk` (это не код), а моноширинный оставить только полю пути. Плейсхолдер пути — обычным трекингом.

### [A22] Кнопки диалога без disabled-валидации (можно создать пустой проект)
- **Severity:** P2
- **Режим:** все
- **Где:** `ProjectListPage.tsx:430` («Добавить» всегда активна), `handleCreate` `:43-54` (полагается на ошибку с сервера)
- **Проблема:** кнопка «Добавить» в диалоге создания не блокируется при пустых «Название»/«Путь» — в отличие от кнопки логина (`isDisabled` по пустым полям). Валидация переложена на сервер: пользователь жмёт «Добавить», ловит ошибку постфактум. Несогласованность с логином и лишний round-trip. Состояние focus у полей диалога тоже не подсвечивается (в отличие от логина с accent-обводкой и glow) — `dialogInput` имеет только `outline: none` без focus-стиля, т.е. фокус вообще не виден.
- **Предложение:** дизейблить «Добавить»/«Сохранить» пока `name`/`path` пусты (как `isDisabled` в логине). Добавить полям диалога focus-состояние (accent-граница + лёгкий glow), консистентно с полями логина (`LoginPage.tsx:128-131`).

### [A23] Дата проекта — `textSecondary` поверх белого мелким, низкий контраст и нет времени
- **Severity:** P3
- **Режим:** все
- **Где:** `ProjectListPage.tsx:304-306` (дата `#756B5E` 12px), `:293-302` (путь `#9A8F7E` 11.5px)
- **Проблема:** под именем три строки нарастающей мелкости: путь 11.5px `#9A8F7E`, дата 12px `#756B5E`. Путь моноширинным 11.5px на белом сложно читать (контраст muted), а дата формата «22 июн.» без года/времени малоинформативна для сортировки свежести. Вертикальный ритм трёх строк (`margin: 3px 0 6px` у пути) выглядит тесновато на скринах.
- **Предложение:** путь — `C.textSecondary` (не muted) для читабельности; дату оставить `textSecondary`, но рассмотреть относительный формат («сегодня», «вчера», «3 дн. назад») как более полезный; выровнять межстрочные отступы (например 2px между всеми тремя строками).
