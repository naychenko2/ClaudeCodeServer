## Находки

### [MD1] Нет ошибок генерации ассистента с разбивкой причин (B4)
- **Severity:** P1
- **Раздел:** B
- **Задумано (макет):** отдельные баннеры под каждую причину `error` у assistant — `rate_limit` (с авто-ретраем и прогрессом), `authentication_failed` («Переавторизоваться»), `billing_error` («Открыть биллинг»), `max_output_tokens` («Продолжить»), `server_error`/`unknown` («Повторить»). Для банковского контура авторизация — ключевой экран.
- **Реализовано:** нет. Бэкенд не парсит поле `error` у `assistant`-сообщения; в протоколе только общий `ErrorMessage(Text)`. Фронт рисует одну плашку «⚠ текст» с кнопкой «Повторить».
- **Расхождение:** пользователь не отличит лимит от просроченного биллинга от обрезанного ответа; нет релевантного действия и авто-ретрая.
- **Рекомендация:** добавить в `ServerMessage` тип `assistant_error { code }`, в `ClaudeSession` извлекать `error` из assistant-блока, на фронте — словарь баннеров по коду с соответствующим действием.

### [MD2] Итог хода без стоимости, токенов и накопления (A3)
- **Severity:** P1
- **Раздел:** A
- **Задумано (макет):** «квитанция» под ответом — `$0.043 за ход`, длительность, под-ходы, `12.4k in · 1.8k out`, накоплено за сессию.
- **Реализовано:** частично. `ResultMessage` несёт `Subtype/DurationMs/NumTurns/Usage`, но **не несёт `total_cost_usd`**, а `ChatItem.result` вообще отбрасывает `usage` (в `useSession.ts` при маппинге `result` поле не переносится). UI = «✓ N шагов · Xс».
- **Расхождение:** ключевая «агентная» ценность (стоимость/токены) не видна, хотя данные частично уже на проводе.
- **Рекомендация:** добавить `totalCostUsd` в `ResultMessage`, пробросить `usage` в `ChatItem.result`, отрисовать карточку-квитанцию; стоимость копить за сессию.

### [MD3] Итоги-ошибки хода не различаются (A4)
- **Severity:** P1
- **Раздел:** A
- **Задумано (макет):** 4 разных ошибочных футера (`error_max_turns` → «Продолжить»/«Лимит», `error_during_execution` → «Повторить ход», `error_max_budget_usd` → сумма, `error_max_structured_output_retries` → техбаннер).
- **Реализовано:** нет. `subtype` доходит до `ChatItem`, но в `ChatItemView` различается только `success` vs прочее (рисует ✗ и те же «N шагов · Xс»). Никаких причин/действий.
- **Расхождение:** при упоре в лимит ходов/бюджет пользователь видит только серое «✗ N шагов» без объяснения и кнопки продолжения.
- **Рекомендация:** ветвление по `subtype` в рендере `result`, ошибочный (красный) вариант футера с текстом причины и релевантным действием.

### [MD4] TodoWrite-план рисуется как безликий tool_use (C5)
- **Severity:** P1
- **Раздел:** C
- **Задумано (макет):** закреплённый чек-лист «ПЛАН 1/3» с галочками completed / пульсирующим in_progress / пустыми pending, обновляется in-place.
- **Реализовано:** нет. `TodoWrite` попадает в общий `case 'tool_use'` — строка «TodoWrite» + первый из `path/command/file_path/pattern` (которого у TodoWrite нет → пустой аргумент). Прогресс задач не виден.
- **Расхождение:** теряется главный индикатор прогресса в длинных задачах; вместо плана — пустая техническая строка.
- **Рекомендация:** спец-обработка `name === 'TodoWrite'`: парсить `input.todos[]`, рисовать чек-лист (желательно закреплённый, а не в потоке).

### [MD5] Нет AskUserQuestion-карточки выбора (D3)
- **Severity:** P1
- **Раздел:** D
- **Задумано (макет):** карточка с вопросом и вариантами A/B (выделенный выбранный), ответ уходит обратно как результат инструмента.
- **Реализовано:** нет. Прилетит как обычный `tool_use` «AskUserQuestion» с сырым аргументом; интерактива нет, ответить из UI нельзя.
- **Расхождение:** модель не может задать уточняющий вопрос с вариантами — ход зависнет/пойдёт по догадке.
- **Рекомендация:** обрабатывать `name === 'AskUserQuestion'` отдельным интерактивным компонентом (как `permission_request`), с обратным каналом ответа.

### [MD6] Тело результата инструмента не показывается совсем (C3)
- **Severity:** P2
- **Раздел:** C
- **Задумано (макет):** карточка переходит в ✓/✗, тело результата (вывод Bash/Read) — **сворачиваемое**.
- **Реализовано:** частично. Привязка по `toolUseId` есть, но в рендере выводится только слово «готово»/«ошибка». `item.result` (текст вывода) нигде не отображается — ни свёрнуто.
- **Расхождение:** для Bash/ошибок сборки пользователь не видит stdout/stderr, только статус.
- **Рекомендация:** добавить раскрываемое тело с моноширинным выводом, особенно для ошибок (`isError`).

### [MD7] Запрос разрешения: нет «всегда разрешать» и не виден tool_input (D1)
- **Severity:** P2
- **Раздел:** D
- **Задумано (макет):** показывать полный `tool_input` (например `$ rm -rf build/`) и чекбокс «Всегда разрешать Bash в этой сессии».
- **Реализовано:** частично. Карточка показывает только `toolName` (без `toolInput`, хотя оно есть в `ChatItem`). Нет «всегда разрешать»; `PermissionDecisionRequest` поддерживает только `allow`/`deny`.
- **Расхождение:** в потенциально опасном контуре пользователь принимает решение, не видя что именно выполнится; нет способа разово доверить инструмент.
- **Рекомендация:** рендерить `toolInput` в моноблоке; добавить опцию «всегда разрешать» с серверным состоянием авто-allow для инструмента в рамках сессии.

### [MD8] Нет типизации инструментов: Edit-diff, Bash-терминал, MCP-значок и пр. (C4)
- **Severity:** P2
- **Раздел:** C
- **Задумано (макет):** разные макеты по `name` — diff-вью для Edit/MultiEdit, терминальная карточка для Bash, выжимка для WebSearch/WebFetch, значок MCP, иконка Skill, превью для Write/Read.
- **Реализовано:** нет (унифицированная строка). Частично спасает отдельный `file_changed` (карточка с +/− и Открыть/Откатить), но он приходит как самостоятельное событие, а не как типизированный `Edit`-tool_use с инлайн-diff.
- **Расхождение:** богатый агентный визуал инструментов отсутствует; самое ценное (diff на телефоне, вывод Bash) не показано в самой карточке инструмента.
- **Рекомендация:** диспетчеризация рендера по `name`; как минимум Bash (вывод) и Edit (diff). MCP/Skill — хотя бы значком.

### [MD9] Task-субагент не вкладывается (C6)
- **Severity:** P3
- **Раздел:** C
- **Задумано (макет):** сообщения субагента вкладываются в раскрываемый блок родительской Task-карточки (дерево), чтобы не засорять ленту.
- **Реализовано:** нет. `parent_tool_use_id` не передаётся протоколом; вложенности нет.
- **Расхождение:** при работе субагентов лента «поплывёт» плоским потоком (сейчас визуально не проявляется, т.к. вложенность вообще не моделируется).
- **Рекомендация:** пробросить `parent_tool_use_id` в `tool_use`/сообщения, группировать под Task-карточкой.

### [MD10] Нет действий «Копировать / Повторить» под сообщением (App-10)
- **Severity:** P3
- **Раздел:** App States · Чат
- **Задумано (макет):** под ответом ассистента — кнопки «Копировать» и «Повторить».
- **Реализовано:** нет. Под `text`-блоком действий нет; «Повторить» доступно только на плашке ошибки.
- **Расхождение:** нельзя быстро скопировать ответ или перезапросить удачный ход.
- **Рекомендация:** добавить hover-панель действий под `text` (копировать текст; повторить последний user-запрос — логика `handleRetry` уже есть).

### [MD11] Граница компакции не отображается (A2)
- **Severity:** P3
- **Раздел:** A
- **Задумано (макет):** тонкая центральная плашка «✦ контекст свёрнут» в длинных сессиях.
- **Реализовано:** нет. `system/compact_boundary` не маппится.
- **Расхождение:** в долгих сессиях пользователь не понимает, что часть истории сжата.
- **Рекомендация:** добавить тип `compact_boundary` и центральный разделитель.

### [MD12] Replay/synthetic пузыри не различаются; нет «Дослать» после прерывания (B1, F)
- **Severity:** P3
- **Раздел:** B / F
- **Задумано (макет):** `isReplay` — приглушённый стиль восстановленной истории; `isSynthetic` — служебная метка; после Interrupt — состояние «Ход остановлен пользователем» с «Дослать».
- **Реализовано:** нет. История при `--resume`/перезагрузке рисуется как обычные сообщения; флаги `isReplay/isSynthetic` не пробрасываются. После `interrupt()` лента просто снимает `isWaiting`, отдельного состояния «остановлено + Дослать» нет.
- **Расхождение:** восстановленная история неотличима от живой; прерывание не оставляет внятного следа в ленте.
- **Рекомендация:** пробросить флаги replay/synthetic и стилизовать; добавить элемент «Ход остановлен» после Interrupt.

**Топ-3 несоответствия:**
1. **MD2 + MD3** — футер итога хода: ни стоимости/токенов при успехе, ни различения ошибок (лимит ходов/бюджет/сбой). Данные частично уже на проводе (`usage` есть, но теряется при маппинге; `subtype` есть, но не используется) — самый дешёвый и заметный выигрыш.
2. **MD4 (TodoWrite)** — план задач рисуется как пустая строка `tool_use`, теряется главный индикатор прогресса.
3. **MD1 (ошибки генерации)** и **MD5 (AskUserQuestion)** — целые задуманные интерактивные/диагностические состояния отсутствуют как класс.
