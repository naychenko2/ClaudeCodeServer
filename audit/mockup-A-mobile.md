### [MA1] Карточка сессии не показывает число сообщений и время
- **Severity:** P1
- **Экран:** Сессии
- **Задумано (макет):** В `Claude Code App.dc.html` карточка сессии содержит превью последнего хода,
  а в нижней строке — иконку «чат» + **число сообщений** (`{{ se.msgs }}`, напр. 14/32/8) и справа
  **относительное время** (`{{ se.time }}`: «сейчас», «1 ч назад», «вчера»).
- **Реализовано:** `components/SessionList.tsx` рендерит имя, превью (`lastMessage`) и бейдж модели.
  Число сообщений и время **не отображаются вообще**. При этом данные ЕСТЬ: в `types/index.ts`
  `Session.messageCount: number`, `createdAt`, `updatedAt` присутствуют — то есть это чисто
  упущение в UI, а не нехватка модели.
- **Расхождение:** Пользователь теряет два ключевых ориентира из задумки — «насколько разговор большой»
  и «когда он был». Карточка беднее задуманной.
- **Рекомендация:** В нижнюю строку карточки добавить иконку-счётчик сообщений (`messageCount`) и
  относительное время от `updatedAt` (формат «N ч/дн назад»), как в макете.

### [MA2] План задач (TodoWrite) не визуализируется в чате
- **Severity:** P1
- **Экран:** Чат
- **Задумано (макет):** В прототипе есть отдельный блок «ПЛАН» с прогрессом `1 / 3` и списком пунктов
  с тремя состояниями: выполнено (зелёная галочка, зачёркнуто), текущее (пульсирующая акцентная точка,
  жирный текст), ожидает (серый контур). Это видный, узнаваемый паттерн отслеживания задач Claude.
- **Реализовано:** В `ChatPanel.tsx` `ChatItemView` обрабатывает kinds: user_message, session_started,
  text, thinking, tool_use, permission_request, file_changed, result, error. Ветки для todo-плана
  **нет**. TodoWrite приходит как обычный `tool_use` и схлопывается в безликую строку «todowrite … готово».
- **Расхождение:** Один из самых характерных элементов UX Claude Code (живой чек-лист плана) полностью
  отсутствует как визуальный блок. Пользователь не видит прогресс по шагам.
- **Рекомендация:** Добавить специальную обработку инструмента TodoWrite в `ChatItemView` — карточку
  «ПЛАН N/M» с пунктами и тремя состояниями (как в макете).

### [MA3] Просмотр файла без нумерации строк
- **Severity:** P2
- **Экран:** Файлы
- **Задумано (макет):** В прототипе просмотр файла (`showFileRead`) и diff показывают код с колонкой
  **номеров строк** слева (`lineNumbers`, моноширинный столбец, серый цвет). В diff номера совмещены
  с маркерами +/−.
- **Реализовано:** `components/FileViewer.tsx` использует `react-syntax-highlighter` без
  `showLineNumbers`. Просмотр кода и diff (ручной рендер `diff.split('\n')`) идут **без нумерации строк**.
  На скриншоте `da-ws-file-mobile.png` — сплошной текст без номеров.
- **Расхождение:** Отступление от задумки; затрудняет ориентирование по коду и обсуждение «строки N».
- **Рекомендация:** Включить `showLineNumbers` в SyntaxHighlighter и добавить номера строк в ручной
  diff-рендер.

### [MA4] Редактирование файла: текстовая кнопка «Править» вместо плавающего FAB-карандаша
- **Severity:** P2
- **Экран:** Файлы
- **Задумано (макет):** В прототипе редактирование файла запускается **плавающей кнопкой (FAB)** —
  акцентный квадрат-карандаш в правом нижнем углу области кода (`showFab` → `toggleEdit`); при
  редактировании FAB заменяется на пару «Отмена»/«Сохранить» там же снизу-справа.
- **Реализовано:** `FileViewer.tsx` в обычном/split режиме показывает **текстовую кнопку «Править»**
  в шапке (`btnPrimary`), а иконку-карандаш — только в fullscreen, тоже в шапке. Плавающего FAB снизу
  нет; «Отмена/Сохранить» при правке тоже в шапке.
- **Расхождение:** Другой паттерн вызова редактирования (шапка vs FAB). На мобиле FAB задуман как более
  крупная и доступная большим пальцем цель.
- **Рекомендация:** Привести к задумке — плавающий FAB-карандаш снизу-справа для входа в режим
  правки, и пара «Отмена/Сохранить» там же при редактировании.

### [MA5] Результаты поиска файлов не показывают папку (директорию)
- **Severity:** P2
- **Экран:** Файлы
- **Задумано (макет):** Результат поиска в прототипе — карточка с иконкой-расширением, **именем файла
  и путём-директорией** под ним (`r.dir`, для корня — «корень»), плюс опц. метка «M».
- **Реализовано:** `components/FileExplorer.tsx` рендерит результаты поиска тем же `renderFileRow`,
  что и дерево, — показывается только имя файла (+ метка M). **Подпись с директорией отсутствует.**
  Видно на `da-ws-files-mobile.png`: четыре строки «design.ts» без указания, в какой папке каждая.
- **Расхождение:** При одинаковых именах файлов в разных папках их невозможно различить — теряется
  смысл результата поиска.
- **Рекомендация:** В режиме результатов поиска под именем файла выводить его директорию (как в макете).

### [MA6] Создание чата: диалог «первого промпта» реализован, но не подключён
- **Severity:** P2
- **Экран:** Сессии
- **Задумано (макет):** В `App States Exploration.dc.html` фрейм 5 «Новая сессия — первый промпт»:
  отдельный шаг с полями «Название (необязательно — авто из первого сообщения)» и «С чего начать»
  (первое сообщение) + кнопка «Создать и начать».
- **Реализовано:** Компонент `components/NewSessionDialog.tsx` точно повторяет этот макет (поля
  Название/Первое сообщение/Режим/Модель, кнопка «Создать и начать»), **но он нигде не используется**
  (grep по `NewSessionDialog` — только определение). Кнопка «+ Новый чат» в `SessionList.createNew`
  создаёт пустую сессию сразу через `api.sessions.create`, без диалога и без поля первого промпта.
- **Расхождение:** Задуманный онбординг-шаг «дай чату имя и первое задание» не доступен пользователю;
  готовый компонент простаивает.
- **Рекомендация:** Подключить `NewSessionDialog` к кнопке «+ Новый чат» (по крайней мере на мобиле),
  либо осознанно отказаться от него и удалить мёртвый код.

### [MA7] Нет inline-баннера «Связь потеряна · переподключаюсь…» поверх чата
- **Severity:** P2
- **Экран:** Чат
- **Задумано (макет):** Фрейм 3 в App States: при потере связи поверх сессии — жёлтый баннер со
  спиннером «Связь потеряна · переподключаюсь…» и ссылкой «Повторить», тело чата приглушено,
  внизу «Ввод недоступен, пока нет связи».
- **Реализовано:** Офлайн обрабатывается иначе: при `!online` composer заменяется заглушкой
  `OfflineComposerStub` («Отправка недоступна офлайн»), кнопки действий скрываются. Отдельного
  **баннера переподключения со спиннером поверх ленты чата нет** (есть `onReconnected` в логике, но
  без выраженного UI-индикатора «переподключаюсь» в чате).
- **Расхождение:** Состояние «временно потеряна связь, идёт реконнект» визуально не отличается от
  «полный офлайн»; пользователь не видит, что система пытается восстановить соединение.
- **Рекомендация:** Добавить верхний баннер переподключения (спиннер + «переподключаюсь…» + «Повторить»)
  на время реконнекта, как в макете.

### [MA8] Голосовой ввод: нет режима записи (дорожка, таймер, отмена/подтверждение)
- **Severity:** P2
- **Экран:** Чат (композер)
- **Задумано (макет):** Фрейм 17 App States «Запись голоса»: во время записи композер показывает
  **живую звуковую дорожку**, **таймер** (`0:06`), кнопку **отмены (✕)** и **подтверждения (✓ →
  расшифровать в текст)**.
- **Реализовано:** `components/Composer.tsx` — простой тоггл микрофона на Web Speech API
  (`SpeechRecognition`): иконка подсвечивается при прослушивании, результат сразу вставляется в поле.
  Нет визуализации дорожки, таймера, явных кнопок отмена/подтверждение.
- **Расхождение:** Реализован «диктовкой на лету», а не задуманным режимом записи с предпросмотром и
  подтверждением. Меньше контроля над результатом (нельзя отменить до вставки).
- **Рекомендация:** Если важно соответствие — добавить визуальный режим записи (таймер + индикатор
  уровня + отмена/подтверждение). Иначе зафиксировать упрощение как сознательное.

### [MA12] Метрика размышления: символы вместо токенов

### [MA14] Пустой проект сразу создаёт сессию — экран «Ещё нет сессий» недостижим
- **Severity:** P3
- **Экран:** Сессии
- **Задумано (макет):** Прототип (пустой список) и фрейм 4 App States показывают состояние
  **«Ещё нет сессий»** с иллюстрацией и кнопкой «Новая сессия» — точку входа в проект без сессий.
- **Реализовано:** `SessionList.tsx` (строки 44–52): если список пуст и онлайн — **автоматически

- **Расхождение:** Реализация подменяет «пустой экран с приглашением» автосозданием — другой UX-выбор.
  Не баг, но отступление от задумки.
- **Рекомендация:** Решить продуктово: либо показывать пустое состояние из макета (создание по кнопке),
  либо явно зафиксировать автосоздание как принятое решение.
