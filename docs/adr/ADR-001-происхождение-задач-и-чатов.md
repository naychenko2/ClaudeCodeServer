# ADR-001: Происхождение задач и чатов (уведомление постановщика + иерархия чатов)

**Статус:** Предложено
**Дата:** 2026-07-23
**Принимающие решение:** Андрей (владелец продукта)
**Реализация:** backend — Денис, frontend — Кира, ревью — Глеб
**Связанные артефакты:** спецификация `.omc/specs/deep-interview-delegated-task-notify.md`

## Контекст

В продукте персоны делегируют задачи друг другу: персона-постановщик A из чата S создаёт
задачу для персоны-исполнителя B (`tasks_create(personaId=B)`), B выполняет её в отдельной
чат-сессии (`TaskExecutionService`) и завершает через `tasks_complete`. Возникли две связанные
потребности:

1. **Уведомление постановщика.** Сейчас по завершении делегированной задачи уведомление уходит
   человеку-владельцу, но персона-постановщик A о завершении не узнаёт и не может продолжить работу.
2. **Иерархия чатов.** В списке чатов дочерние чаты-исполнители лежат плоско вперемешку с
   родительскими; нет визуальной связи «из какого чата какие задачи-чаты порождены».

Обе потребности опираются на один отсутствующий факт — **какой чат породил задачу**. Ключевые
силы и ограничения:

- **Инвариант per-owner изоляции** — все данные токенятся сервисным JWT владельца; межвладельческих
  утечек быть не должно.
- **Директива `Session.cs:5`** — происхождение чата вычисляется из `TaskId`/`AutomationRuleId`,
  отдельно не хранится, «чтобы не было второго источника истины» (реализовано как computed `Session.Origin`).
- **Персона активна только внутри хода сессии** — нет фонового агента, «получающего сообщения»;
  «уведомить персону» = либо ход в её чате, либо notification человеку.
- **Существующая инфраструктура:** движок проактивности (`PersonaAutomationService` +
  `TaskStatusTriggerSource`) уже умеет будить персону по смене статуса задачи; `Session.TaskId`
  уже хранится и есть в DTO; `TaskExecutionService` уже уведомляет человека по завершению.
- **Стоимость:** старт claude CLI ~15с + токены на каждый агентский ход — «раздал 10 задач» не
  должно молча превращаться в 10 платных ходов.

## Решение

Ввести **единый факт происхождения** и построить обе фичи на нём, не создавая новых процессов
и не нарушая инвариант единственного источника истины.

1. **Хранимое поле — только `TaskItem.SourceSessionId`** (чат, породивший задачу). Заполняется
   через проброс `TASKS_SESSION_ID` в `BuildTurnMcpConfig`; tasks-server кладёт его в тело POST
   при создании задачи. Обслуживает **обе** фичи.
2. **Родитель чата — вычисляемый, не хранимый.** `parentSessionId` резолвится в Session-DTO из
   `Session.TaskId → Task.SourceSessionId` (по образцу computed `Session.Origin`).
3. **Уведомление постановщика — два этапа, дефолт L0** (бесплатное notification без хода);
   активная реакция ходом (L1/L2) — осознанный опт-ин через движок проактивности.
4. **Иерархия чатов — представление на фронте** по вычисленному `parentSessionId` (тумблер вида,
   не фича-флаг).

## Рассмотренные варианты

### Решение A — Канал уведомления постановщика

#### A1: Новый MCP-сервер «watch task»
| Параметр | Оценка |
|----------|--------|
| Сложность | Высокая |
| Стоимость | Новый node-процесс + канал |
| Соответствие инвариантам | Дублирует событийный движок |

**Плюсы:** изолированная ответственность.
**Минусы:** второй источник истины по состоянию задач; ещё один процесс; слать *некому* (адресат
уведомления всегда человек). Отклонён.

#### A2: Существующий движок проактивности (`TaskStatusTriggerSource`)
**Плюсы:** переиспускает троттлинг, quiet-hours, `HourlyCap`, Gate/Work; ноль новых процессов.
**Минусы:** `Gate` в движке — тоже платный ход (не «дешёвое уведомление»); реакция в отдельном
чате правила; требует существования правила.

#### A3 (выбран): Двухэтапная схема с дефолтом L0
- **Этап 1 (MVP, L0):** `TaskExecutionService` по завершении задачи с `SourceSessionId`/постановщиком
  шлёт notification от лица A со ссылкой на исходный чат. Движок правил не задействован.
- **Этап 2 (L1/L2):** активный постановщик через движок проактивности (`createdByMe`-фильтр,
  доставка ходом, автосоздание правила).

**Плюсы:** исходная проблема закрыта минимальным Этапом 1; дефолт не жжёт платные ходы; инвариант-риск
границы чатов уезжает в опциональный Этап 2.
**Минусы:** два кодовых пути (notification и ход); понятие уровней L0/L1/L2 надо объяснить в UI.

### Решение B — Модель «родитель чата»

#### B1: Хранимый `Session.ParentSessionId` (денормализация)
| Параметр | Оценка |
|----------|--------|
| Сложность | Низкая |
| Устойчивость | Переживает удаление задачи |
| Соответствие инвариантам | Третий денормализованный источник, против `Session.cs:5` |

#### B2: Фронт джойнит стор задач (`chat.taskId → task.sourceSessionId`)
**Минусы:** каждому списку нужен загруженный стор задач; при удалении задачи иерархия молча
разваливается. Отклонён.

#### B3 (выбран): Вычисляемый `parentSessionId` в Session-DTO
**Плюсы:** ноль новых хранимых полей в `Session`; единственный источник истины; фронту приходит
готовое поле (не нужен стор задач); соответствует директиве `Session.cs:5`.
**Минусы:** при удалении задачи связь теряется — чат всплывает в корень (принято осознанно:
чаты-исполнители завершаются, а не удаляются).

### Решение C — Join-барьер «когда готовы все N»
Отклонён: требует новой сущности «пакет делегирования» и персистентного состояния «k из N».
Вместо этого — по-задачно, постановщик агрегирует остальные свои задачи через `tasks_list` в ходе.

### Решение D — Autostart делегированной задачи
Выбран **промпт-подход**: инструкция персоне-постановщику вызвать `tasks_execute` (переиспускает
гейт `TASKS_EXECUTE = agentDepth < 1`). Программный autostart в контроллере отклонён (создание
задачи не должно неожиданно порождать сессию).

## Анализ компромиссов

- **Единое поле vs два (`SourceSessionId` против `CreatedInSessionId` + `ParentSessionId`).**
  Слияние в один хранимый факт устраняет дубли и делает вторую фичу почти бесплатной надстройкой
  над первой. Цена — обе фичи связаны релизно (Этап 1 уведомления и фундамент иерархии идут вместе).
- **Вычисляемый родитель vs хранимый.** Выбор в пользу инварианта единственного источника истины;
  платим потерей связи при (редком) удалении задачи. Обратимо: если удаление окажется частым —
  добавить хранимое поле точечно, не переделывая остального.
- **Дефолт L0 vs Gate.** Дешёвый дефолт защищает от лавины платных ходов ценой того, что «из коробки»
  постановщик лишь уведомляется, а не действует. Апгрейд — один переключатель уровня правила.

## Последствия

**Становится проще:**
- Постановщик узнаёт о завершении своих делегированных задач (Этап 1) минимальными правками.
- Иерархия чатов получает фундамент (`SourceSessionId` + `TASKS_SESSION_ID`) бесплатно от Этапа 1.
- Ноль новых процессов и хранимых точек истины по родителю чата.

**Становится сложнее:**
- Два кодовых пути уведомления (notification L0 / ход L1-L2).
- Понятие уровней L0/L1/L2 требует объяснения в UI автоматизаций.

**К чему придётся вернуться:**
- **Инвариант-риск (только Этап 2):** доставка auto-хода в живой пользовательский чат размывает
  границу «чат правила ↔ рабочий чат» — под наблюдением архитектора; митигация маркером в ленте.
- Если удаление задач окажется частым — пересмотреть вычисляемый родитель в пользу хранимого.
- Join-барьер — если появится спрос на оркестрацию «когда готовы все».
